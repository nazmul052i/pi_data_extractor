The Data Conditioning Dialog

The Data Conditioning dialog is used to specify the limits that define bad data samples and to specify averaging techniques to account for laboratory data samples that are acquired infrequently with respect to other input samples.

Note:  All processing of bad data occurs before averaging computations are performed. Also, all limit values (for example, cutoffs, thresholds) represent absolute values. Data samples that exceed limits in either a positive or negative direction cause the corresponding limit-related action to be taken.

The Variables section contains information about model variables.

Note:  Use the horizontal scroll bar to view columns that are not visible.

  • The Name column indicates each variable’s tag name.

  • The Type column indicates each variable’s current type: Ind (Independent) or Dep (Dependent).

  • The Cut column indicates the number of data samples that have been cut (i.e., that violate upper and/or lower data cutoff limits) for the corresponding tag.

  • The Bad column indicates the number of data samples that are considered bad (See Processing Bad Input Data in IQmodel Reference).

  • The Minimum and Maximum columns contain the current minimum and maximum values for the corresponding tag.

  • The Lower Cutoff and Upper Cutoff columns contain the current lower and upper data cut limits that defined bad data for the corresponding tag. A blank field indicates that the corresponding limit is not enabled. (See The Data Conditioning Properties Dialog below).

  • The Flatline column indicates the period (in absolute time units) that the variable’s value must be below the Flatline Threshold for an abnormal constant value to be detected (if Flatline detection is enabled).

  • The Threshold column to the right of the Flatline column indicates the threshold value that must be exceeded by at least one sample in the set of samples collected during the Flatline Period to prevent a Flatline fault (if Flatline detection is enabled).

  • The Spike column indicates the period (in absolute time units) that sample values must remain above the corresponding Spike Threshold value in order for IQmodel to consider that the data values are normal (in other words, that the observed spike actually represents the process moving to another operating level).

  • The Threshold column to the right of the Spike column indicates the threshold value which, when exceeded on a sample-to-sample basis, triggers a Spike fault and causes the Spike Period timer to start (if Spike detection is enabled).

  • The Independent: Variables edit box indicates the current number of model input (independent) variables. The corresponding Samples edit box indicates the number of data samples available (per independent variable).

Note:  Any variable whose status was set to Off in the Data Specification dialog is not available in this and subsequent model building steps; a variable’s absence is reflected in the number of available independent variables.

  • The Dependent: Variables edit box contains the tag name of the current model output (dependent) variable. The corresponding Samples edit box indicates the number of dependent variable data samples available.

  • The Training and Validation edit boxes indicate the number of samples used for model training and model validation, respectively. The number of samples for training and validation are both zero when the Data Conditioning dialog first opens.

  • The Model Status: edit box indicates the current model status (Invalid , Ready , or Out of Date ). The model status is Invalid when the Data Conditioning dialog first opens.

The Processing section allows you to specify data conditioning processing options. Additional information is provided in the Processing Bad Input Data and Averaging Input Data sections below.

  • The Replace Bad Input Samples check box indicates whether or not bad samples should be replaced (this box must be checked to enable parameter fields in the Replace Bad Input Samples section of the Conditioning Expert Parameters dialog).

  • The Time-Average Inputs check box indicates whether or not input samples should be averaged (this box must be checked to enable the parameter fields in the Time Average Input section of the Conditioning Expert Parameters dialog).

  • The Validate Model check box indicates whether or not some data samples should be reserved to validate the model (this box must be checked to enable fields in the Validate Model section of the Conditioning Expert Parameters dialog).

To change variable properties, to delete a variable from the list of input variables available for building the model, or to restore variables to the list, right-click on the name of a variable in the Variables section. The pop-up menu shown below will be displayed:

Deleting and Restoring Conditioned Variables

To delete a variable (i.e., make to it unavailable for use in later model building steps), select Delete from the pop-up menu.

Note:  Deleting a variable does not remove it or its associated data from the underlying model file; it simply keeps the variable from being used in subsequent steps of the model building process.

To restore all variables that have been deleted in this model building step, select Restore from the pop-up menu.

Note:  Even though individual variables may be deleted, all previously deleted variables are restored as a group. To restore only some deleted variables, you must restore them all and then re-delete those you actually want deleted.

If you select Properties from the pop-up menu, you can change properties for the highlighted variable(s), as described in The Data Conditioning Properties Dialog.

The Validation section allows you to specify validation parameters:

  • The Validate Model check box indicates whether a portion of the input data should be reserved to validate the model. When Validate Model is checked, the Validation Rate: and Random Seed: edit boxes are made available.

  • The Validation Rate: edit box allows you to specify the portion (as a decimal fraction) of the dataset you want to reserve for validation.

  • The Random Seed: edit box allows you to specify a seed value for the random number generator used to select input data samples for validation.

    Note:  Every component of IQmodel that relies on a random number generator allows you to specify a seed for the generator used by that particular component.

  • The Training edit box indicates the number of dependent variable data samples that will be used for training (building) the model.

  • The Validation edit box indicates the number of dependent variable data samples that will be used for validating the model.

The Data Conditioning Properties Dialog

The Data Conditioning Properties: dialog allows you to specify the limit values that serve to identify outlier (i.e., bad) data points, and to specify whether or not the limits should be applied to the variables listed in the Variables section.

The Title bar of the Properties: dialog shows the name of the selected variable.

The Cutoffs section allows you to specify how IQmodel handles data outliers.

  • If a check box contains a check mark, the corresponding limit is enabled. IQmodel uses the contents of the Value: edit box as the limit threshold.

Note:  Bad data samples can also be marked using the Input Data grid view.

The Flatline Detection section allows you to specify whether IQmodel should detect abnormal series of constant input data.

  • If the Apply Detection: check box contains a check mark, the contents of the Period: and Threshold: edit boxes are used to identify Flatline faults (extended series of constant values).

  • The Period: box indicates the length of time (in absolute time units) that changes in the variable’s values must remain below the value in the Threshold: edit box to constitute a Flatline fault. IQmodel ensures that the period is a multiple of the Base Time of the model.

The Spike Detection section allows you to specify whether IQmodel should detect abnormal changes in the magnitude of data samples.

  • If the Apply Detection: check box contains a check mark, the contents of the Period: and Threshold: edit boxes are used to identify Spike faults (rapid sample-to-sample changes in data values).

  • The Period: box indicates the length of time (in absolute time units) that the variable’s values must remain above the value in the Threshold: edit box before IQmodel considers that the detected spike is no longer a fault, but rather that the process has moved to a different operating level. In other words, the Period: value in effect determines the number of consecutive Spike faults that IQmodel will report. IQmodel ensures that the period is a multiple of the Base Time of the model.

  • If Spike Detection is enabled, as soon as the sample-to-sample change in value exceeds the specified threshold, a Spike fault is reported. If the absolute value of subsequent samples remains above the value corresponding to the (sample + threshold) value that generated the fault for the time specified in the Period: box, samples collected after the period expires will no longer be marked bad as Spike faults.
