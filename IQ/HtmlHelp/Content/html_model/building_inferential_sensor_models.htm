Building Models with Aspen IQmodel

Aspen IQmodel provides a comprehensive set of capabilities and features that support building robust sensor validation and/or inferential sensing empirical models derived from plant data.

For models built using IQmodel, it is also important to ensure that model training data in fact accurately reflect the target process over the operating range expected when the model is deployed online. In practical terms, this means ensuring that the model is trained with a span of data sufficiently long that all normal operating conditions, including acceptable extremes, are included. The perceived quality of input data should also be reflected in the limits used to identify abnormal data values.

When you wish to build a sensor validation models, the recommended approach is to first build an inferential sensor model to identify which sensors are most important for the inferential model. Then, concentrate on building a good sensor validation model for each of the important physical sensors used by the inferential sensor model. The goal is to build a sensor validation model that has good identifiability for the physical sensor of interest.

To judge whether or not a sensor validation model has good identifiability:

  1. Examine the Structured Residual Transforms | Sensor Values grid. This matrix maps sensor values to normalized structured residual values, which have an expected value of 0 and a standard deviation of 1. The diagonal of this matrix contains values close to zero that reflect the insensitivity of each transform to its corresponding sensor.

  2. Examine the column corresponding to the sensor you are most interested in identifying. If all the elements of the column (other than the diagonal element) are large (for example, greater than 3), then there should be good identifiability. In general, the larger the value of the smallest (non-diagonal) element in the column, the better the identifiablity. Conversely, if any of the non-diagonal elements in the column are less than 1, you should try removing some of the physical sensors (tags) from the model in order to improve identifiablity.

  3. Once you think you have constructed a good sensor validation model, test it in Run Mode by inducing artificial faults. A model that exhibits good identifiability should be capable of detecting even small faults.

  4. If a supposedly good model fails to detect drift, bias, or complete failure, look at the Structured Residuals graph. During the fault, the structured residual for the bad sensor should be less than its threshold, while all other sensors should be above their thresholds. If necessary, adjust the identification confidence percentage in the Expert dialog to increase or decrease the threshold. Similar comments apply to the Generalized Likelihood Ratio graph. For precision loss faults, look at the Normalized Cumulative Variance graph.

Model Identification Batch Processing

When the Batch option is selected, as illustrated below, the dialog boxes for each of the Model Identification steps can be opened, parameters set, and then closed, without causing IQmodel to perform any processing.

After all needed parameters for all Model Identification steps have been set, clicking the Run Batch button causes IQmodel to perform, in sequence, the calculations for each step and to pass relevant results to the next step. For large models that use genetic algorithm search strategies, it is usually more efficient to set the parameters for all steps before beginning execution, since the genetic algorithm is computationally intensive and can take some time to complete.

Canceling and Restarting IQmodel Search Calculations

If search strategy computations for any of the Model Identification steps are interrupted (i.e., the Cancel button on the associated progress dialog is clicked), IQmodel retains the state of the computation. When the model is saved, the state information is also stored in the model file. If IQmodel is terminated then restarted at a later time, the model can be loaded, the appropriate Model Identification dialog opened, and search calculations restarted at the point where they were originally interrupted. If stored state information is available in a model, clicking the Search button in the relevant Model Identification dialog will bring up a pop-up message box. Options in the pop-up message box allow the user to specify either that search calculations should start over, or that calculations should resume based on state information saved with the model.

After models have been built and trained, further offline analysis and tuning can be performed using datasets other than the original training dataset. Through Expert Parameters dialogs associated with every major model building step, parameters that govern the basic structure and features of the current model can be adjusted. As each model building step is completed, a variety of grid and graphical views is provided to permit quick visual assessments of model output resulting from actions at the step.

Finally, when model characteristics and performance are judged to completely and accurately reflect the process being modeled, IQmodel can export a model file specifically designed for online deployment through other members of the Aspen IQ product family.
