Lab Update (LBU) Module

See Also

Creating a Lab Update

Even the best inferential model will eventually drift if no model updating is applied (based on either feedback measurements from the laboratory, on-line analysis, or both).

Aspen IQ supports laboratory and analyzer updating, and can track the drift of the analyzer from the laboratory if both measurements are available.

When the model prediction output for the lab bias update, LBUDYN, is set to 1, the unbiased dynamic model output (UAZ) will be used to calculate the lab bias. Otherwise, the steady-state model output (UPR) is used in the lab bias calculation. If the lab data used to derive the prediction model contained process dynamic information, then a dynamic model output should be used. When the lab frequency is low, the steady-state model output should be used for the lab bias update.

The lab update (LBU) module uses the information gathered by the lab data collection (LDC) module. The LDC module stores the lab value along with an array of unbiased and biased prediction values and total percentage steady state values at sample time in the LDC history file. The LDC module also sets both the CHAINFLAG entry and an internal chainflag to one 1 after a new lab value is stored. The LBU module will detect the change in the internal chainflag entry and reset its value to 0. The externally exposed CHAINFLAG parameter is for use with external programs and will not be reset by Aspen IQ.

The LBU module also has a lab bias validation chain flag (LBULABFLAG). This flag is set to 1 after the LBU module successfully validates a lab sample. This flag will be set even if the validation status is BAD. Aspen IQ will not reset this parameter to 0. Similarly the LBU module has a lab bias update chain flag (LBUBIASFLAG). This flag is set to 1 after the LBU module successfully updates the lab bias. Aspen IQ will not reset this parameter to 0 either.

The LBU module reads the new lab value from the LDC history file. For each new lab value, an array of unbiased and biased prediction values and total percentage steady-state values starting at PREDSTMSTR and ending at PREDETMSTR are retrieved from the LDC history file. The data retrieval start and end times are calculated using the prediction history offset, PREDOFST, and prediction history interval, PREDITVL, specifications as follows:

PREDSTMSTR = Lab Sample Time - PREDOFST - PREDITVL

PREDETMSTR = Lab Sample Time - PREDOFST

Note that the LDC module uses a similar calculation to determine the data retrieval range when reading from the data collection (DC) history file. The offset and interval specifications for HISTOFST, HISTITVL, PREDOFST, and PREDITVL must be such that the data requested by the LBU module is a subset of the data stored by the LDC module.

If an analyzer update (AZU) module is defined for the same IQ, the LBU module will read the validated (raw analyzer + lab offset) analyzer values, AZUVAZ, from the AZU history file which were stored, starting at ANZRSTMSTR and ending at ANZRETMSTR. The start and end times are calculated using the analyzer history offset, ANZROFST, and the analyzer history interval, ANZRITVL, specifications as follows: 

ANZRSTMSTR = Lab Sample Time - ANZROFST - ANZRITVL

ANZRETMSTR = Lab Sample Time - ANZRDFST

The number of unbiased prediction values retrieved from the LDC history file for the current lab sample is shown in UPRNVALS. The UPRNGOOD entry indicates the number of unbiased prediction values with a "Good" status. The UPRSDEV entry indicates the standard deviation of the unbiased prediction values. The average of the unbiased prediction values retrieved for each lab sample is stored in the PLBUIUPR entry. The status of the average unbiased prediction depends upon the number of values with "Good" status and the DCPCTGOOD entry specified in the Config section. The most recent average unbiased prediction value with a good status is stored in the LBUIUPR entry.

There are similar entries for the biased prediction values, namely, BPRNVALS, BPRNGOOD, BPRSDEV, PLBUIBPR, and LBUIBPR. The entries for the total percentage steady state values are LSSNVALS, LSSNGOOD, LSSSDEV, PLBUILSS, and LBUILSS. The entries for validated analyzer values read from the AZU history file are VAZNVALS, VAZNGOOD, VAZSDEV, PLBUIVAZ, and LBUIVAZ.

After collecting the prediction, steady state, and analyzer values corresponding to the new lab sample, the LBU module will perform the following computations

  • Lab validation.

  • Lab bias calculation.

  • Analyzer drift calculation.

The lab bias calculations will not be performed for an IQ whose lab bias update method, LBUPDMETH, is configured as "Validation Only". This option is useful in a testing phase where the inferential sensor model does not exist as of yet, so bias updating is not appropriate. The analyzer drift calculations will not be performed for an IQ that does not contain an analyzer update module.

Lab Validation

The validation process screens incoming lab data for minimum, maximum, rate of change limits, and outliers.

A lab value (PLBULAB) is flagged as bad in the validation message (PLBUVLDNUM) if:

  • It exceeds the lab maximum limit, LVMAXLIM.

  • It is lower than the lab minimum limit, LVMINLIM.

  • The steady-state value at sample time, PLBUILSS, is lower than the lab minimum percentage steady-state limit, LVMINPCTSS and
    LVMINPCTSS is greater than zero.

  • The lab value has moved lower in one iteration than the lab decreasing rate of change limit, LVROCLIMD and LVROCLIMD is greater than zero.

  • The lab value has moved higher than the lab increasing rate of change limit, LVROCLIMI and LVROCLIMI is greater than zero.

  • If the lab sample is older than the maximum number of hours for lab bias update, LVMAXHOURS, then the validation message number for last good lab value, LBUVLDNUM, is flagged as bad status.
    The old lab sample detection is bypassed when LVMAXHOURS is less than or equal to zero and LVMAXHOURS is greater than zero.

  • If absolute difference between the lab value, PLBULAB, and the biased prediction at lab sample time, PLBUIBPR exceeds the lab outlier error limit, LVOLRLIM and LVOLRLIM is greater than zero. The lab sample is considered an outlier if this is the only failed check.

Each subsequent outlier in the same direction will cause the lab outlier counter, (LVOLRCNTI for increasing or LVOLRCNTD for decreasing) in that direction to be incremented and the current lab sample will be rejected.

When the counter for number of outliers in the same direction reaches the respective limit set in LVOLRLIMI and LVOLRLIMD, then all of the outliers are determined to be valid values and the statuses for each value will be changed to "Good." If a "Good" lab is received before the counter limit is reached, then previously flagged outliers will stay "Bad." If the outlier count limit is set to zero, then the "Bad" statuses will always remain and not be flagged "Good."

The new lab bias value, LBIASNEW, is available for validation via the validation section.

Lab Bias Calculation

The lab bias update calculations use the previous lab sample values, PLBULAB, and their corresponding biased and unbiased prediction values, PLBUIBPR and PLBUIUPR, to calculate a raw bias value, PLBIASRAW. The raw bias value is calculated as the difference between an average of previous lab values, PLBUAVGLAB, and an average of corresponding unbiased prediction values, PLBUAVGUPR.

PLBIASRAW = PLBUAVGLAB - PLBUAVGUPR

The number of lab values averaged to calculate PLBUAVGLAB depends on the LBUNUMVALS entry and may vary depending on the bias update method specified in the LBUPDMETH entry.

Note:  In version V8.8 and later, the lab bias calculation uses any number of lab values from one (1) to LBUNUMVALS. This change allows the predictions to be corrected immediately after the first good lab sample is entered. There is no longer a requirement that LBUNUMVALS samples must be entered before the first bias update occurs.

The bias update methods currently supported in Aspen IQ are:

  • Traditional

  • Cusum

  • SCORES

These methods are explained in later in this section.

A fraction of the change in prediction bias recommended by the raw bias value, PLBIASRAW, is applied to the existing prediction bias and stored in the new lab bias value, PLBIASNEW. The fraction is specified using the LBIASFRAC entry:

PLBIASNEW = PREDBIAS + (PLBIASRAW – PREDBIAS) * LBIASFRAC

The lab values and the corresponding prediction values used in lab bias calculation must have a "Good" status. If there are insufficient numbers of previous lab samples with good status, then the bias update calculation will fail and the bias update message, PLBIASNUM, will be set accordingly.

A successful bias calculation will cause the values of the LBIASNEW, LBIASRAW, LBUAVGLAB, LBUAVGUPR, and LBUAVGBPR entries to be updated from the corresponding PLBIASNEW, PLBIASRAW, PLBUAVGLAB, PLBUAVGUPR, and PLBUAVGBPR entries for the current sample. The bias update time entry, LBIASTMSTR, will be updated to the current time.

If there are no successful bias updates for a period exceeding the number of hours specified in LVMAXHOURS, then the lab bias status, LBIASPNTSTS, will be set to "Bad."

Relative Lab Bias Calculation

In certain circumstances it may be desirable to have IQ apply the lab bias as a percentage of the current unbiased prediction rather than as an absolute value. This feature is enabled with the LBUPDREL flag.

With the relative lab bias option enabled, IQ will calculate the percentage bias and then apply a constant percentage over the prediction's range. This option may be useful with processes that have transitions over a wide range of prediction values. This option will not work well with prediction values that are near zero because the relative bias will be disproportionately large.

The percentage update is stored in the parameter LBIASNEWPCT. Once a relative bias is calculated the absolute value of the bias is stored in PLBIASNEW. Although LBIASNEWPCT is not historized, it is easily calculated from the previous lab data as 100*PLBIASNEW/PLBUAVGLAB. With the relative lab bias option enabled, LBIASNEWPCT will remain constant between lab updates and LBIASNEW will change each execution of the prediction module as the unbiased prediction changes.

Traditional update method

The first update technique is the traditional approach (commonly used in the industry). With this approach, you can pre-determine the number of validated lab samples to be used for the lab update.

In processes where infrequent samples are taken and significant unmeasured disturbances are frequent, lab updates should be performed with nearly every available lab sample.

This frequency allows you to keep the inferential sensor model tracking the lab as consistent as possible. As the frequency of updates increases, the chances of inducing the laboratory error into the inferential sensor measurement increases. You can temper this effect by performing an update based on the average of a number of previous lab samples. You set the desired number of lab samples for the given inferential sensor.

The number of lab samples and their corresponding predictions to use for the update are set by the LBUNUMVALS parameter.

Note:  In version V8.8 and later, LBUNUMVALS is treated as the maximum number of lab samples to use in a lab bias update calculation. The actual number of samples used can vary from one (1) to LBUNUMVALS.

CUSUM update method

The second update technique is based on a CUSUM approach. This approach does not pre-determine the number of lab samples used for update. Instead, it calculates the number based on some simple statistical rules.

The error in prediction is calculated as the difference between the lab value, PLBULAB, and the corresponding biased prediction, PLBUIBPR. The prediction error for the most recent lab sample is stored in the LBUIPRERR entry.

This error is added to the accumulated error of previous lab samples, LBUACUMERR. A CUSUM update counter, LBUACUMCNTR, is incremented each time a prediction error associated with a new lab sample is added to the accumulated error, LBUACUMERR.

The lab bias update counter, LBUACUMCNTR, is clamped at the high limit specified for the number of samples used for the lab bias update, LBUNUMVALS, and is reset to 0 when the absolute accumulated error, LBUACUMERR, exceeds the lab bias update CUSUM error limit, LBUCUSUMLIM.

When the accumulated error is greater than the LBUCUSUMLIM threshold, a lab bias update occurs. The number of lab samples used in this raw lab bias update calculation is determined as the minimum of either of the following:

  • The number of lab samples required to provide the necessary total error that meets the CUSUM error limit, LBUCUSUMLIM.

    --or--

  • The number of lab samples equal to LBUNUMVALS.

An update occurs when the LBUCUSUMLIM threshold is met, even if there are fewer than LBUNUMVALS good samples available.

If LBUCUSUMLIM is exceeded using number of samples less than LBUNUMVALS, then the number of samples required to meet the threshold is used for the lab bias update calculation.

If LBUCUSUMLIM is exceeded using number of samples greater than LBUNUMVALS, then LBUNUMVALS is the number of samples used in the raw bias update calculation.

Note: In version V8.8 and later, no automatic bias update occurs when the number of samples reaches LBUNUMVALS because the bias update requires that the accumulated error is greater than or equal to LBUCUSUMLIM first. Additionally, there is no longer a requirement that LBUNUMVALS samples must be entered before the first bias update occurs.

The CUSUM update technique will behave differently than previously described if the most recent lab sample that contributed to LBUCUMERR is older than LVMAXHOURS. Since it is older than LVMAXHOURS it is thrown out but it's contribution to the LBUACUMERR remains. This will set the Lab Bias Update Status to “Sample too old” and in the next cycle the Update Source Method will be from “Lab” to “Manual” with the Manual bias set to the last lab bias value.

Additional new lab samples will lead to changes in LBUCUMERR even if LBUCUMERR is greater than LBUCUSUMLIM and LBUACUMCNTR will increment by 1 even if LBUACUMCNTR is already greater than or equal to LBUNUMVALS. No bias updates will occur until the number of lab samples is equal to LBUNUMVALS, within a time frame of LVMAXHOURS, and LBUCUMERR is greater than LBUCUSUMLIM. When this occurs then the Lab Bias Update Status will be set back to “Good”, the Update Source Method will be set to “Lab” and the CUSUM bias update will occur as normally.

Due to this behavior it's possible for the accumulated error of the previous lab samples that have now gone bad to be responsible for the LBUCUMERR to be greater than LBUCUSUMLIM and if the accumulated error was just considered for the good lab samples within LVMAXHOURS then the update would not have occurred. Note that when the average lab value is calculated for the bias update it is only considering the good lab samples within the LVMAXHOURS time frame and not considering the older lab values that contributed to LBUCUMERR.

SCORES update method

The final update option is a SCORES approach. When a new lab sample is available, a mean and standard deviation is calculated for the user-defined population, LBUNUMVALS.

The current score, LBUCURRSCOR, is calculated as the difference between the current lab value, LBULAB, and the average of past LBUNUMVALS lab values divided by the standard deviation of lab values.

The average of the lab values is stored as LBUAVGLAB. The current score, LBUCURRSCOR, is copied to the previous score, LBUPREVSCOR, before updating the current score based on the new lab sample.

The Scores lab update method calculates a new lab bias value when the accumulated score, LBUACUMSCOR, exceeds the set limit, LBUSCORLIM. The accumulated score is reset to zero whenever the current score changes signs. To accumulate scores, the individual error, LBUIPRERR, must be in the same direction long enough for the accumulated score to equal or exceed the limit.

Note:  In version V8.8 and later, there is no longer a requirement that LBUNUMVALS samples must be entered before the first bias update occurs.

Analyzer Drift update

When both lab and analyzer measurements are available for an IQ, it is critical that the lab and analyzer values used to adjust the prediction bias are at the same level, especially when switching from an analyzer update source to lab or vice versa.

Frequently, the analyzer value has a somewhat constant offset from the lab value. The AZU module contains an entry, AZULABOFST, which is added to the raw analyzer value, AZURAZ. The AZULABOFST entry can be used to account for the offset between the lab and the analyzer. The result of AZURAZ + AZULABOFST is stored in the AZUVAZ entry and referred to as the validated analyzer value.

As mentioned earlier, the LBU module will read validated analyzer values at lab sample time and store the average of these values in the PLBUIVAZ entry for that sample.

The analyzer drift is calculated as the difference in average of previous lab values, DRFAVGLAB, and the average of their corresponding validated analyzer values, DRFAVGVAZ. The number of lab samples used for averaging is specified by DRFTNUMVALS.

PANZRDRIFT = PDRFAVGLAB - PDRFAVGVAZ

The lab values and the corresponding validated analyzer values used in drift calculation must have a "Good" status. If there are an insufficient number of previous lab samples with "Good" status, then the drift calculation will fail and the drift update message, PDRFTNUM, will be set accordingly.

A successful drift calculation will cause the values of the ANZRDRIFT, DRFAVGLAB, and DRFAVGVAZ entries to be updated from the corresponding PANZRDRIFT, PDRFAVGLAB, and PDRFAVGVAZ entries for the current sample. The drift update time entry, DRFTTMSTR, will be updated to the current time.

If there are no successful drift updates for a period exceeding the number of hours specified in LVMAXHOURS, then the analyzer drift status, DRIFTPNTSTS, will be set to "Bad."

Lab update initialization

The lab bias may be initialized using the lab bias initialization flag (LBUINIT). This entry will set the lab bias (LBIASNEW) to zero and mark all previous lab samples as obsolete. The LBIASNEW value will remain at zero until a new sample arrives. At that point LBIASNEW will be set to the difference between the lab value and the unbiased prediction. Once a valid lab sample is entered and the LBIASNEW value is successfully calculated, the LBUINIT flag will be reset to zero. Samples marked as obsolete will remain so and LBUNUMVALS lab samples will be needed for calculation of subsequent bias updates.

Lab data variability

Lab data, as with all measurements, has some inherent variability and measurement noise. The lab variability is highly dependent upon the repeatability and reproducibility of the laboratory tests themselves. Factors such as which lab apparatus was used, which lab personnel performed the test, the condition of the lab sample before being sent to laboratory, etc. can all play a major factor in the lab variability.

This lab variability can create significant issues when updating the inferential sensor. AspenTech recommends that you determine the lab variability with a series of blind tests at the normal laboratory facility and possibly other "neutral" laboratories as well, to gain a more thorough understanding of the lab variability for the samples in question.
