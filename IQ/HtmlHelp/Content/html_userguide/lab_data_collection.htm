Lab Data Collection (LDC) Module

Creating a Lab Data Collection

Transform Overview

To perform a laboratory update, you must first set up a lab data collection. The lab data collection module stores laboratory data with synchronized process data and corresponding inferential sensor predictions to the LDC history file.

When a new laboratory value is available, a range of desired process data and the corresponding inferential sensor predictions at the sample time are retrieved from the process data collection files. The data is stored in the LDC history file. The maximum number of lab samples stored in the LDC file is specified using the LBUMAXREC entry.

The resulting LDC history file is then used by the lab update (LBU) module, or the contents can be extracted using IQextract, for model development or maintenance.

The first configuration required for the lab data collection section is the how the lab data sample taken flag is configured. IQpredict supports five methods of determining when a when the lab value is taken.

A sample taken flag, SAMPFLAG, (1 = sample taken, 0 = not taken) can be configured by itself or in conjunction with a string that holds a time stamp, STKTMSTRIN. When used by itself, the flag will store the current date/time stamp in the "sample taken" time stamp.

When used with the string, the time stamp will be used rather than the current value. The STKTMSTRIN entry supports both ASC (ASCII string tagname) and TIME (timestamp tagname) Cim-IO source types.

For those systems where strings cannot be easily passed, six integers (month, day, year, hour, minute, second) are used instead. These values and the flag are treated in a similar manner as the strings.

After a sample is taken, SAMPFLAG is reset to 0 and the lab available status message, NEWLABSTSNUM, is set to "Waiting." The timestamp is then written as a string to STKTMSTROUT.

If another sample is taken before the lab value for this sample is entered into the system, then the status for the new sample is set to "In Lab." All subsequent samples will say "In Lab" until the sample that is waiting has a lab value entered for it. Only one sample at a time will display "Waiting."

If the sample taken time predates an existing sample taken time, then the sample will be rejected. If a sample taken time is more than four hours in the future, this sample will also be rejected. To view reasons for a sample rejection, see the Messages view, which is in Aspen Production Control Web Server.

The LDCPRDST entry contains the average biased prediction for the lab sample with the "Waiting" status.  This value is the sum of the current PREDBIAS and the average unbiased steady-state or dynamic prediction near the sample time (PLDCPRDST). PLDCPRDST is calculated using the interval and offset variables specified in the LBU module (if available) so that it will correspond to the impending bias update calculation. LDCPRDST is available to the operators as they enter the lab sample value and therefore can provide insight as to the direction the lab bias is likely to move once the sample is entered. 

You can cancel the "Waiting" sample by setting the lab sample canceled flag, CNCLFLAG, to 1. When CNCLFLAG is set, the waiting sample will be set to "Cancelled", and CNCLFLAG is then reset to zero.

The total number of "Waiting" and "In Lab" samples is stored in NUMWAIT, and is compared to the maximum number of samples waiting in lab, MAXWAIT. If NUMWAIT equals MAXWAIT, then any sample taken after that point will be rejected. To reduce NUMWAIT, a lab value must be entered for the "Waiting" sample and subsequently for the "In Lab" variables (or else these samples must be cancelled).

LDC History File

The lab data collection history file stores the following information for each inferential quality:

LB

Raw Lab Value

LPV

Lab Process Value

PV

Process Value

LUPR

Unbiased Prediction

LBPR

Biased Prediction

LSS

Total Percentage Steady-State

When the lab value is available, it is entered into NEWLAB and the lab available flag, AVLFLAG, is set to 1

The Define PV is used to specify a subset of the data collection points, which will be stored in the LDC history file along with the laboratory value. For each lab value, an array of PV values, starting at STORSTMSTR and ending at STORETMSTR, are retrieved from the DC history file. The data retrieval start and end times are calculated using the PV history offset, HISTOFST, and PV history interval specification.

STORSTMSTR = Lab Sample Time HISTOFST HISTITVL

STORETMSTR = Lab Sample Time HISTOFST

If a prediction (PR) module is defined for the IQ, then the LDC module will also store an array of biased and unbiased prediction values starting at STORSTMSTR and ending at STORETMSTR in the LDC history file. The LBUDYN entry is used to specify dynamic versus steady state prediction values. The LBU module to calculate lab bias later retrieves these prediction values.

The LDC module will also retrieve an array of "Total Percentage Steady-State" values starting at STORSTMSTR and ending at STORETMSTR from the DC history file and store it in the LDC history file along with a lab sample. The steady-state values are later retrieved by the LBY module and used to validate the lab sample.

The Define Lab PV button is used to specify additional lab values that are associated with the lab sample for the given IQ. The current value is read from the LPVPNTVAL entry and stored in the LDC history file at the same time when the lab value is read from the NEWLAB entry. This is helpful when multiple lab values, such as a complete D-86 or D-2889 distillation test results are to be collected with the property in question.

When the lab value, predictions, and additional process tags are written to the lab data collection file, the new lab sample status, NEWLABSTSNUM, is changed from Waiting to Stored. The timestamps for when the sample was taken and when it was available are stored into the new lab value available time, NEWLABSTMSTR, and new lab value sample time, NEWLABATMSTR, respectively.

For systems where passing strings is not permitted, the new lab status message is also stored as an integer in the new lab sample status message number, NEWLABSTSNUM, parameter. The Lab Data Collection module will set the CHAINFLAG parameter to 1 when a lab sample is available. Similarly, an internal chainflag will be set to 1 so that the Lab Update Module will know to process the new lab sample. The CHAINFLAG parameter is available so that external applications can be made aware of the new lab sample. Aspen IQ will not reset CHAINFLAG to 0.

LDC Input Entries

Note:  The AVLFLAG, CNCLFLAG and SAMPFLAG entries will be reset to zero by the LDC module if the keyword for these entries is LOCAL or RDWRT.

The following entries provide input values to lab data collection modules:

AVLFLAG

New lab

CNCLFLAG

Lab Sample Cancelled Flag

PV History Interval

HISTOFST

PV History Offset

LBUDYN

Prediction used for Lab Bias Update

MAXWAIT

Maximum Number of Samples Waiting in Lab

NEWLAB

New Lab Value

SAMPFLAG

Sample Taken Flag (STIMEOPT = 1,3, or 5 only)

STIMEOPT

Sample Time Option

If STIMEOPT is 2 or 3, then the sample time is read from the following entry:

STKTMSTRIN – Sample Taken Time Input

If STIMEOPT is 4 or 5, then the sample time is read from the following entries:

STKNDAY – Sample Taken Day

STKNHOUR – Sample Taken Hour

STKNMIN – Sample Taken Minute

STKNMON – Sample Taken Month

STKNSEC – Sample Taken Second

STKNYEAR – Sample Taken Year
